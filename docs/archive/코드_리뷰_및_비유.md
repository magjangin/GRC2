# 코드 리뷰 및 전체 로직 비유 설명

## 🎵 전체 시스템 비유: 콘서트 공연 제작 시스템

이 모드는 **음악 게임을 위한 커스텀 콘서트 제작 시스템**입니다. 마치 무대 뒤에서 모든 것을 조율하는 프로듀서처럼 작동합니다.

---

## 📋 전체 흐름 비유

### 1단계: 공연 준비 (게임 시작 → 모드 초기화)

**비유**: 콘서트장에 도착한 프로듀서가 무대 뒤에서 모든 준비를 시작합니다.

```
🎬 SceneDetector.OnInitializeMelon()
   ↓
🧷 SteamManifestLocker.LockManifest()
   → "공연장 운영 규칙 고정" (게임 업데이트로 무대 구조가 바뀌는 걸 방지)
   ↓
🎼 MusicInjector.Initialize()
   → "공연 프로그램 편성" (곡 선택 리스트에 커스텀 공연을 끼워넣을 준비)
   ↓
📁 hwa 폴더 생성 (공연 자료 보관함)
   ↓
🔍 GameTypeInspector.SearchFairyModeNotesManager()
   → 게임의 "무대 장치(노트 매니저)" 파악 (cFairyModeNotesManager 타입/필드 확인)
   ↓
🧰 BmsNoteConverter.Initialize()
   → 내부에서 GameTypeLoader.Initialize()를 호출해 NoteCreateData/Enum 타입 준비
   ↓
📝 BmsParser.ParseBmsFile()
   → BMS 파일 읽기 (악보 읽기)
   → BPM 정보 추출 (템포 파악)
   → 노트 데이터 파싱 (연주 지시 읽기)
   ↓
🔧 Harmony 후킹 설정
   → 게임의 "중요한 순간"에 끼어들 준비
   → 마치 "무대 뒤에서 조용히 대기하는 스태프"
```

**핵심**: 게임이 시작되자마자 모든 준비를 끝냅니다. 사용자는 아무것도 모르지만, 뒤에서는 이미 모든 것이 준비되고 있습니다.

---

### 2단계: 악보 해석 (BMS 파싱)

**비유**: 복잡한 악보를 읽어서 게임이 이해할 수 있는 형식으로 번역합니다.

```
📄 BMS 파일 (원본 악보)
   ↓
🔍 BmsParser.ParseBmsFile()
   ├─ BPM 정보 추출 (#BPM, #BPMXX)
   │  → "이 부분은 빠르게, 저 부분은 느리게"
   ├─ 노트 데이터 파싱 (채널 11-16, 18)
   │  → "1번 레인에 터치, 2번 레인에 홀드..."
   ├─ HoldNoteProcessor.MatchHoldNotes()
   │  → 홀드 시작(02)과 끝(19) 매칭
   │  → "이 홀드는 2초 동안 누르고 있어야 해"
   ├─ FairyNoteProcessor.MatchFairyNotes()
   │  → 페어리 시작(11-18)과 끝(1A-1B) 매칭 (같은 IsLeft 그룹 내 FIFO, 레인 제한 없음)
   │  → "이 페어리는 회전하면서 3초 동안"
   └─ CalculateTime()
      → tick을 초로 변환 (BPM 변화 고려)
      → "정확히 1.234초에 이 노트가 나타나야 해"
   ↓
📦 BmsNote 리스트 (번역된 악보)
```

**핵심**: BMS 파일은 게임이 이해할 수 없는 형식이므로, 게임이 이해할 수 있는 형식으로 완전히 변환합니다.

---

### 3단계: 게임 형식으로 변환 (BmsNote → NoteCreateData)

**비유**: 번역된 악보(BmsNote)를, 게임이 읽을 수 있는 “표준 연주 지시서(NoteCreateData)”로 다시 작성합니다.

여기서 `NoteCreateDataBuilder`는 마치 **공연 진행표를 작성하는 행정 담당자**처럼 동작합니다.
- 게임 버전에 따라 “지시서 양식(생성자/필드)”이 조금씩 다를 수 있으니,
- 먼저 **맞는 양식(생성자)을 여러 버전으로 시도**하고,
- 안 맞으면 **빈 양식(기본 생성자)을 꺼내 수기로 필드를 채운 뒤**,
- 마지막으로 “시간표(perfectSample)가 제대로 적혔는지” 검수까지 합니다.

```
📦 BmsNote 리스트 (번역된 악보)
   ↓
🔄 BmsNoteConverter.ConvertBmsNotesToNoteCreateData()
   ├─ 시간 순으로 정렬
   │  → "연주 순서대로 정리"
   ├─ NoteCreateDataBuilder.CreateNoteCreateData()
   │  → 각 노트를 게임 형식으로 변환
   │  ├─ 1) 시간표 작성: perfectSample = Time * 48000
   │  │  → "정확히 몇 샘플에 등장해야 하는지"
   │  ├─ 2) 양식 선택(생성자 탐색): NoteConstructorHelper.TryFindConstructor()
   │  │  → "이 버전 양식? 저 버전 양식?" (2~6개 파라미터 시도)
   │  │  → 실패하면 기본 생성자(빈 양식)로 생성
   │  ├─ 3) 수기 기입(필드 채우기): FieldAccessHelper + NoteFieldInitializer
   │  │  ├─ laneLeftRightID / subLaneID / noteTypeID
   │  │  ├─ directionIndex
   │  │  │  → FairyEnd는 CENTER_TOP(깊이용), 그 외는 BMS 방향/레인 규칙
   │  │  ├─ turnDireciton (페어리 회전)
   │  │  │  → FairyEnd(1A/1B) 또는 EndNote.Direction을 우선 반영
   │  │  └─ noteSize / slideEndFlickDirection / Boolean 기본값들
   │  └─ 4) 검수: perfectSample이 0/null이면 보정 + 경고 로그
   ├─ HoldNoteProcessor.ProcessHoldEndNotes() (구현: HoldNoteProcessor.ConnectNodes.cs)
   │  → 홀드 끝 노트를 connectNodeDataArray에 연결
   │  → "이 홀드의 끝 부분은 여기에 연결되어 있어"
   └─ FairyNoteProcessor.ProcessFairyEndNotes() (구현: FairyNoteProcessor.ConnectNodes.cs)
      → 페어리 끝 노트를 connectNodeDataArray에 연결
      → "이 페어리의 끝 부분은 여기에 연결되어 있어"
   ↓
🎯 NoteCreateData 배열 (게임이 이해할 수 있는 연주 지시서)
```

**핵심**: 게임은 `NoteCreateData`만 읽습니다. 그래서 “양식 선택(생성자)”과 “수기 기입(필드 세팅)”을 조합해 어떤 게임 버전에서도 최대한 안전하게 지시서를 만들어냅니다. (역변환 검색은 Dictionary 캐싱으로 빠르게)

---

### 4단계: 무대 뒤에서 조정 (Harmony 후킹)

**비유**: 게임이 노트를 생성하려고 할 때, 무대 뒤에서 조용히 우리의 노트로 교체합니다.

```
🎮 게임: "노트를 생성하겠어!" (createAllNote 호출)
   ↓
🪝 NoteArrayHooks.CreateAllNotePrefix() (가로채기!)
   ├─ TryInjectBmsNotes()
   │  → (가드) CustomAssetManager.ShouldInjectCustomContent()가 false면 주입을 건너뜀
   │     - 주입 금지 씬(SoundPlayerScene, MoviePlayer_MovieSelect)에서는 항상 건너뜀
   │     - 커스텀 차트 미선택이면 건너뜀
   │  → (주입) 변환된 BMS 노트를 mFairyNoteCreateDataArray에 주입
   │  → "우리 커스텀 차트로 교체!"
   └─ (JSON 덤프는 `NoteArrayJsonDumper`가 기본 비활성화라 파일이 생성되지 않음)
   ↓
🎮 게임: "알겠어, 이 노트들로 연주하겠어!"
   → 게임은 자신이 원래 차트를 사용한다고 생각하지만,
   → 실제로는 우리의 커스텀 차트를 사용함
```

**핵심**: Harmony는 게임의 메서드를 "가로채서" 우리 코드를 먼저 실행시킵니다. 게임은 아무것도 모르지만, 실제로는 우리의 노트를 사용합니다.

**현재 코드 기준 보충(중요):**
- 문서에 자주 등장하는 “원본/주입 차트 JSON 덤프”는 `GRC2/Harmony/Hooks/NoteArrayHooks.cs`에서 `NoteArrayJsonDumper.Initialize(noteFolderPath, isEnabled: false)`로 **기본 비활성화**되어 있어, 기본 설정으로는 파일이 생성되지 않습니다.
  - 덤프가 필요하면 위 `isEnabled`만 `true`로 바꾸면 됩니다.

---

### 5단계: 배경 영상/음악 교체 (BGA/BGM 주입)

**비유**: 무대의 배경 영상과 음악을 우리가 준비한 것으로 교체합니다.

#### BGA (배경 영상) 주입

```
🎬 BgaInjector.TryInjectBgaCoroutine()
   ├─ VideoPlayer 찾기 (모든 활성화된 VideoPlayer)
   │  → "무대의 영상 재생 장치 찾기"
   ├─ 파일 크기 확인 및 타임아웃 조정
   │  → "큰 파일이면 더 오래 기다려야 해"
   ├─ VideoPlayer.url 설정 및 Prepare
   │  → "우리 영상 파일로 설정"
   └─ VideoPlayer.Play()
      → "재생 시작!"
```

#### BGM (배경 음악) 주입

```
🎵 BgmInjector.TryInjectBgmCoroutine()
   ├─ cBGMBeatManager 인스턴스 찾기 (캐싱)
   │  → "음악 재생 장치 찾기"
   ├─ BgmLoader.LoadAndInjectAudioClip()
   │  ├─ UnityWebRequestMultimedia.GetAudioClip()
   │  │  → "오디오 파일 로드"
   │  ├─ cBGMBeatManager.setClip()
   │  │  → "우리 음악으로 교체!"
   │  ├─ VerifyInjection()
   │  │  → "제대로 교체되었는지 확인"
   │  └─ BgmFinishTimeManager.SetFinishTime()
   │     → "이 음악의 길이만큼 게임이 진행되어야 해"
   └─ requestPlayAudio()
      → "재생 시작!"
```

**핵심**: 게임의 원본 BGA/BGM을 우리가 준비한 파일로 교체합니다. 플레이 씬에서만 주입을 시도합니다.

---

### 6단계: 게임 종료 시간 조정 (무한 플레이 모드)

**비유**: 원래 게임은 120초 후에 끝나지만, 우리 음악은 180초이므로 게임이 끝나지 않도록 조정합니다.

```
⏰ BgmGameEndMonitor.AdjustMusicDataForBgmLength() (구현: BgmGameEndMonitor.Adjustments.cs)
   → mRythmGameMusicData의 4개 필드 조정
   ├─ musicFadeOutEndSample: BGM 길이 (샘플)
   │  → "음악이 끝나는 시점"
   ├─ musicFadeOutStartSample: BGM 길이 - 1초
   │  → "음악 페이드 아웃 시작 시점"
   ├─ screenFadeOutStartSample: BGM 길이 - 1.5초
   │  → "화면 페이드 아웃 시작 시점"
   └─ screenFadeOutEndSample: BGM 길이
      → "화면 페이드 아웃 종료 시점"
   ↓
🎮 게임: "아, 이 음악은 180초구나! 그럼 180초까지 진행하겠어!"
   → 원래는 120초에 끝났지만, 이제 180초까지 진행됨
```

**핵심**: 게임이 조기에 종료되지 않도록 4개 필드를 모두 조정해야 합니다. 하나라도 빠지면 게임이 원래 시간에 끝납니다.

---

### 7단계: 게임 종료 모니터링

**비유**: 음악이 끝나면 자동으로 공연을 마무리합니다.

```
🎵 BgmMonitorCoroutine.MonitorBgmFinishCoroutine()
   ├─ 0.1초마다 BGM 재생 시간 확인
   │  → "음악이 얼마나 재생되었는지 체크"
   ├─ BGM 길이 도달 시
   │  → "음악이 끝났어!"
   └─ requestCommonRythmGameEnd() 호출
      → "결과 화면으로 전환!"
```

**핵심**: 주입된 BGM 길이를 정확히 참조하여, 음악이 끝나면 자동으로 결과 화면으로 전환합니다.

## 🏗️ 주요 컴포넌트 비유

### 1. **SceneDetector** - 프로듀서
- **역할**: 전체 공연을 기획하고 조율하는 프로듀서
- **비유**: 콘서트장에 도착해서 모든 준비를 시작하는 사람
- **책임**: 초기화, 씬 감지, 전체 흐름 제어

### 2. **BmsParser** - 악보 해석가
- **역할**: BMS 파일을 읽어서 이해 가능한 형식으로 변환
- **비유**: 복잡한 악보를 읽어서 연주 지시로 번역하는 사람
- **책임**: BPM 추출, 노트 파싱, 시간 계산

### 3. **BmsNoteConverter** - 번역가
- **역할**: BmsNote를 NoteCreateData로 변환
- **비유**: 한 언어로 된 악보를 다른 언어로 번역하는 사람
- **책임**: 형식 변환, 정렬, 끝 노트 연결

### 4. **NoteCreateDataBuilder** - 연주 지시서 작성자
- **역할**: 각 노트를 게임 형식으로 변환
- **비유**: “공연 진행표”를 게임 표준 양식으로 작성하는 행정 담당자  
  - 양식(생성자)이 버전마다 다를 수 있어 여러 양식을 먼저 시도하고, 안 되면 빈 양식에 수기로 채움
- **책임**: `NoteConstructorHelper`로 생성자 시도 → `FieldAccessHelper`/`NoteFieldInitializer`로 필드 채움 → perfectSample 검수/보정 → 역변환 검색 캐싱

#### 4-1. **NoteConstructorHelper** - 양식(생성자) 선택 담당
- **역할**: NoteCreateData “양식 버전(생성자 시그니처)”을 2~6개 파라미터 순서로 시도
- **비유**: “이 양식으로 제출해 주세요”가 버전마다 달라서, 맞는 양식을 찾아주는 담당
- **책임**: `TryFindConstructor()`로 생성자 탐색/호출, 필요 시 `LogNoteCreateDataConstructorsAndFields()`로 디버그 목록 출력

#### 4-2. **NoteFieldInitializer** - 수기 기입 규정 담당
- **역할**: directionIndex/turnDireciton/noteSize/slideEndFlickDirection/Boolean 기본값을 규칙대로 채움
- **비유**: 빈 양식에 “어디에 무엇을 써야 하는지” 규정을 가진 담당
- **책임**: FairyEnd(1A/1B) 특수 처리, 페어리 회전 방향 우선순위, 기본값 세팅

### 5. **HarmonyHookManager** - 무대 뒤 스태프
- **역할**: 게임의 메서드를 가로채서 우리 코드 실행
- **비유**: 무대 뒤에서 조용히 조정하는 스태프
- **책임**: 후킹 설정, 메서드 가로채기

### 6. **NoteArrayHooks** - 노트 교체 담당
- **역할**: 게임이 노트를 생성할 때 우리 노트로 교체
- **비유**: 무대에 올라갈 악보를 바꿔치기하는 사람
- **책임**: 노트 주입, (선택) JSON 덤프

### 6-1. **CustomChartHandler** - 공연 프로그램 감별사 (Services)
- **역할**: 곡 선택 UI에서 “지금 커스텀 공연인가?”를 감지하고, 맞으면 아트워크/BGM 로드를 트리거
- **비유**: 공연장 입구에서 티켓(곡 정보)을 보고 ‘특별 공연(커스텀 차트)’인지 판별하는 스태프
- **책임**: `mCurrentDispData`/`mMusicSelectData` 등의 필드를 캐싱해 리플렉션 비용을 줄이면서 커스텀 차트 감지

### 6-2. **ResultSceneInjector** - 커튼콜 연출 담당 (결과 씬)
- **역할**: 결과 씬(`RythmGameResultScene`)에서 커스텀 아트워크와 난이도 표시를 주입
- **비유**: 공연이 끝난 뒤 커튼콜 화면에 포스터(아트워크)와 등급(난이도)을 걸어주는 담당
- **책임**: 결과 씬 오브젝트 탐색 후 스프라이트 적용/표시 요소 정리(LV 부품 숨김 등)

### 7. **BgmBgaInjector** - 미디어 담당
- **역할**: BGA/BGM 파일을 찾아서 주입
- **비유**: 배경 영상과 음악을 준비하는 사람
- **책임**: 파일 검색, 주입 시도, 코루틴 관리

### 8. **BgmGameEndMonitor** - 시간 관리자
- **역할**: 게임 종료 시간을 조정하여 무한 플레이 모드 구현
- **비유**: 공연 시간을 조정하는 스케줄러
- **책임**: 4개 필드 조정, 게임 종료 시간 관리

### 9. **FieldAccessHelper** - 정보 수집가
- **역할**: 리플렉션을 통한 필드 접근을 캐싱
- **비유**: 자주 찾는 정보를 메모해두는 사람
- **책임**: 필드 정보 캐싱, 빠른 접근

### 10. **EnumValueHelper** - 번역 사전
- **역할**: Enum 값 파싱 결과를 캐싱
- **비유**: 자주 찾는 단어를 사전에 메모해두는 사람
- **책임**: Enum 값 캐싱, 빠른 검색

## 🔧 코드 품질 개선

### 1. **중복 코드 제거**

**비유**: 같은 일을 여러 곳에서 반복하는 대신, 한 곳에서 관리하는 것.

#### 씬 처리 로직 통합
- `HandleFairyModeScene()`과 `HandlePlayMovieScene()`의 중복 로직을 `HandlePlayScene()` 공통 메서드로 통합
- **효과**: 코드 라인 수 약 50줄 → 30줄로 감소, 유지보수성 향상

#### 끝 노트 체크 로직 통합
- 홀드와 페어리 끝 노트 체크 로직을 `CheckMissingEndNotes()` 공통 메서드로 통합
- **효과**: 코드 라인 수 약 100줄 → 50줄로 감소, 일관된 에러 메시지

#### Prefix 메서드 통합
- `CreateAllNotePrefix()`와 `LoadFairyNoteDatasJsonToArrayPrefix()`의 중복 로직을 `TryInjectBmsNotes()` 공통 메서드로 통합
- **효과**: 에러 처리 일관성 개선, 스택 트레이스 자동 포함

#### 생성자 찾기 로직 리팩토링
- 반복적인 생성자 시도 로직을 `TryFindConstructor()` 메서드로 통합
- 생성자 시그니처를 배열로 정의하여 순회 방식으로 개선
- **효과**: 코드 라인 수 약 200줄 → 80줄로 감소, 가독성 향상

### 2. **에러 처리 일관성 개선**

**비유**: 모든 직원이 같은 방식으로 문제를 보고하는 것.

#### ErrorLogger 헬퍼 클래스
- 모든 catch 블록에서 일관된 에러 로깅 패턴 사용
- `LogException()`: 예외를 표준 형식으로 로깅 (Error 레벨)
- `LogWarning()`: 치명적이지 않은 오류를 경고 레벨로 로깅
- InnerException 자동 처리
- 컨텍스트 정보 및 스택 트레이스 자동 포함

#### 표준화된 에러 처리
- 약 27개의 catch 블록을 `ErrorLogger` 사용으로 통일
- 에러 메시지 형식 일관성 확보
- **효과**: 유지보수성 향상 (에러 로깅 로직이 한 곳에 집중), 디버깅 용이성 향상

## ⚠️ BMS 작성 주의사항

### 홀드/페어리 끝 노트 필수 배치

**비유**: 홀드/페어리 노트는 시작과 끝이 쌍을 이루어야 합니다. 끝 노트 없이는 노트가 제대로 처리되지 않습니다.

**중요:**
- 홀드 노트(02) 또는 페어리 노트(11-18)를 사용하는 경우, **반드시 해당 끝 노트를 배치해야 합니다.**
- 끝 노트를 배치하지 않으면:
  - **BMS 노트 주입이 거부됩니다** (안전을 위해)
  - 로그에 "BMS 파일을 다시 확인해보세요!" 메시지가 출력됩니다
  - 게임은 원본 차트로 플레이됩니다

**올바른 BMS 작성:**

**페어리 노트:**
```
#00111: 0011000000000000  // 페어리 시작(11) ✅
#00211: 001A000000000000  // 페어리 끝(1A=Left 턴) ✅ 필수!
```

**홀드 노트:**
```
#00111: 0002000000000000  // 홀드 시작(02) ✅
#00211: 0019000000000000  // 홀드 끝(19) ✅ 필수!
```
