# 홀드 노트 처리 가이드

이 문서는 BMS 파일의 홀드 노트(02-19)를 게임의 Hold 노트로 변환하는 과정과 관련 문제 해결 방법을 설명합니다.

**최신 업데이트 (2025년)**:
- **매칭 로직 혁신**: `StartNote`와 `EndNote` 필드를 추가하여 **상호 참조(Reference) 기반 매칭** 구현 (부동소수점 오차 원천 차단)
- 기존의 시간/레인 기반 검색은 참조가 없는 레거시 데이터에 대한 폴백(Fallback)으로만 사용되어 **매칭 정확도 100% 보장**
- 홀드 노트 및 페어리 노트 모두 동일한 참조 시스템 적용
- 기존 개선 사항(직접 찾기, perfectSample 비교 등)은 폴백 로직으로 유지되어 안전장치 역할 수행
- **BPM 기반 동적 시간 허용 범위** 및 범위 검색 추가 (폴백 사용 시)

**중요(현재 코드 기준)**:
- 홀드/페어리를 포함한 **BMS 노트 주입은 `NoteArrayHooks.TryInjectBmsNotes()`에서 `CustomAssetManager.ShouldInjectCustomContent()` 결과를 기준으로 제어됩니다.**
- 즉, **커스텀 차트가 선택되어 있고, 주입이 금지된 씬이 아닐 때만** BMS 노트가 주입됩니다.
- `CustomAssetManager.IsSceneWhereInjectionDisallowed()`에서 `SoundPlayerScene`, `MoviePlayer_MovieSelect`는 항상 주입이 차단되는 **주입 금지 씬**으로 취급됩니다.
- `hwa` 루트에 BMS가 있더라도, 커스텀 차트를 선택하지 않은 일반 곡이나 주입 금지 씬에서는 BMS 노트가 주입되지 않습니다.

## 홀드 노트 개요

### BMS 형식
- **02**: 홀드 시작 (Hold Start)
- **19**: 홀드 끝 (Hold End)

### 게임 형식
- **NoteType.Hold**: 홀드 노트 타입
- **connectNodeDataArray**: 홀드 끝 노트를 담는 배열

### BMS 값 → 노트 타입 매핑

`BmsNoteDataParser.GetNoteType()`에서 BMS 값을 노트 타입으로 변환합니다:

| BMS 값 (16진수) | 노트 타입 | 설명 |
|----------------|----------|------|
| `01` | `Touch` | 일반 터치 노트 |
| `02` | `Hold` | 홀드 시작 ⚠️ |
| `03-0A` | `Flick` | 플릭 노트 (8방향) |
| `11-18` | `Fairy` | 페어리 노트 (8방향) |
| `19` | `HoldEnd` | 홀드 끝 ⚠️ |
| `1A-1B` | `FairyEnd` | 페어리 끝 (1A=Left 턴, 1B=Right 턴) |

**⚠️ 중요**: 
- 홀드 시작 노트는 반드시 `02` 값을 사용해야 합니다. `03-0A` 값을 사용하면 플릭 노트로 인식됩니다!
- 홀드 끝 노트는 반드시 `19` 값을 사용해야 합니다. `03-0A` 값을 사용하면 플릭 노트로 인식되어 `connectNodeDataArray`에 플릭 노트가 생성됩니다!

## 노트 타입 판별 로직

### BmsNoteDataParser.GetNoteType() 동작

```csharp
private static NoteType GetNoteType(int value, out NoteDirection? direction)
{
    direction = null;
    
    // 우선순위 1: 특정 값들
    if (value == 0x01) return NoteType.Touch;      // touch
    if (value == 0x02) return NoteType.Hold;       // Hold ⚠️
    if (value == 0x19) return NoteType.HoldEnd;   // Hold end
    if (value == 0x1A || value == 0x1B) return NoteType.FairyEnd;
    
    // 우선순위 2: 범위 체크 (플릭)
    if (value >= 0x03 && value <= 0x0A)  // ⚠️ 02가 아닌 03-0A는 플릭으로 인식!
    {
        direction = GetFlickDirection(value);
        return NoteType.Flick;
    }
    
    // 우선순위 3: 범위 체크 (페어리)
    if (value >= 0x11 && value <= 0x18)
    {
        direction = GetFairyDirection(value);
        return NoteType.Fairy;
    }
    
    return NoteType.Touch;  // 기본값
}
```

**핵심 포인트:**
- `02`는 **반드시** `NoteType.Hold`로 인식됨
- `19`는 **반드시** `NoteType.HoldEnd`로 인식됨
- `03-0A`는 **반드시** `NoteType.Flick`로 인식됨
- BMS 파일에서 홀드 시작 노트에 `03-0A` 값을 사용하면 플릭으로 잘못 인식됨
- BMS 파일에서 홀드 끝 노트에 `03-0A` 값을 사용하면 플릭으로 잘못 인식되어 끝 노트가 플릭으로 생성됨 ⚠️

### EnumValueHelper.GetNoteTypeId() 동작

파싱된 `NoteType`을 게임의 `noteTypeID` Enum 값으로 변환합니다:

```csharp
public static object GetNoteTypeId(NoteType noteType)
{
    switch (noteType)
    {
        case NoteType.Touch:
            return GetEnumValue(GameTypeLoader.NoteTypeIdEnum, "Touch");
        case NoteType.Hold:
        case NoteType.HoldEnd:
            return GetEnumValue(GameTypeLoader.NoteTypeIdEnum, "Hold");  // ⚠️
        case NoteType.Flick:
            return GetEnumValue(GameTypeLoader.NoteTypeIdEnum, "Flick");
        case NoteType.Fairy:
        case NoteType.FairyEnd:
            return GetEnumValue(GameTypeLoader.NoteTypeIdEnum, "Fairy");
        default:
            return GetEnumValue(GameTypeLoader.NoteTypeIdEnum, "Touch");
    }
}
```

**핵심 포인트:**
- `NoteType.Hold` → `noteTypeID = "Hold"`
- `NoteType.HoldEnd` → `noteTypeID = "Hold"` (끝 노트도 Hold로 설정됨)
- `NoteType.Flick` → `noteTypeID = "Flick"`
- 만약 홀드 끝 노트가 파싱 단계에서 `NoteType.Flick`으로 잘못 인식되면, `noteTypeID`도 `"Flick"`으로 설정됨
- 이 경우 `connectNodeDataArray`에 플릭 노트가 생성되어 홀드 노트가 제대로 작동하지 않음 ⚠️

## 처리 과정

### 1단계: BMS 파싱 단계 (`BmsParser.ParseBmsFile`)

먼저 BMS 파일을 파싱하여 `BmsNote` 리스트를 생성합니다:

```csharp
// BMS 파일에서 노트 데이터 파싱
var notes = ParseBmsFile(filePath);

// 각 노트의 타입은 GetNoteType()에서 결정됨
// 02 → NoteType.Hold
// 19 → NoteType.HoldEnd
```

**⚠️ 주의**: 
- 이 단계에서 홀드 시작 노트에 `02`가 아닌 값이 사용되면 잘못된 타입으로 인식됩니다!
- 이 단계에서 홀드 끝 노트에 `19`가 아닌 값(특히 `03-0A`)이 사용되면 플릭으로 인식되어 끝 노트가 플릭으로 생성됩니다!

### 2단계: 홀드 노트 매칭 (`HoldNoteProcessor.MatchHoldNotes`)

```csharp
// 같은 레인에서 02(시작)와 19(끝)을 매칭
// 노트 채널만 필터링 (11, 12, 14, 15, 16, 18)
var validChannels = BmsNoteDataParser.ChannelToLaneMap.Keys.ToHashSet();
var holdStarts = notes.Where(n => n.Type == NoteType.Hold && validChannels.Contains(n.Channel)).ToList();
var holdEnds = notes.Where(n => n.Type == NoteType.HoldEnd && validChannels.Contains(n.Channel)).ToList();

foreach (var start in holdStarts)
{
    // 같은 레인 + 같은 방향(IsLeft)에서 가장 가까운 19(Hold end) 찾기
    var end = holdEnds
        .Where(e => e.Lane == start.Lane && e.IsLeft == start.IsLeft && e.Tick > start.Tick)
        .OrderBy(e => e.Tick)
        .FirstOrDefault();

    if (end != null)
    {
        // Duration 계산 (Tick 단위로 저장, 나중에 Time으로 변환됨)
        start.Duration = end.Tick - start.Tick;
    }
}
```

**핵심 포인트:**
- **채널 필터링 개선**: `ChannelToLaneMap`을 사용하여 모든 노트 채널(11, 12, 14, 15, 16, 18)을 포함
- 같은 레인(`Lane`)에서 매칭
- 시작 노트보다 나중에 오는 가장 가까운 끝 노트를 찾음
- `Duration`을 Tick 단위로 계산하여 저장

### 3단계: 노트 변환 단계 (`BmsNoteConverter.ConvertBmsNotesToNoteCreateData`)

```csharp
// 홀드/페어리 끝 노트는 별도 처리
if (bmsNote.Type == NoteType.HoldEnd)
{
    holdEndNotes.Add(bmsNote);
    continue; // 메인 배열에 추가하지 않음
}
```

**핵심 포인트:**
- 홀드 끝 노트(19)는 메인 노트 배열에 추가하지 않음
- 별도 리스트(`holdEndNotes`)에 저장하여 나중에 처리

### 4단계: 끝 노트 연결 단계 (`HoldNoteProcessor.ProcessHoldEndNotes (구현: HoldNoteProcessor.ConnectNodes.cs)`)

**개선된 매칭 로직 (2025년 최신 버전):**
1. **상호 참조(Reference) 기반 매칭 (우선순위 1)**:
   - `BmsParser` 단계에서 이미 연결된 `StartNote` 참조를 사용하여 O(1)로 즉시 Start 노트를 찾습니다.
   - 시간/레인 비교가 필요 없으므로 부동소수점 오차나 중복 노트 문제에서 완전히 자유롭습니다.
   - 대부분의 경우 이 경로를 통해 처리됩니다.

2. **기존 검색 로직 (우선순위 2 - 폴백)**:
   - 참조가 없는 경우(레거시 데이터 등), 아래의 기존 로직을 안전장치(Fallback)로 사용합니다.

**기존 매칭 로직 (2024년 버전 - 현재는 폴백으로 작동):**

```csharp
// 1. allBmsNotes에서 홀드 시작 노트를 직접 찾아서 noteList와 매칭
var holdStartBmsNotes = allBmsNotes.Where(n => n.Type == NoteType.Hold && n.Duration > 0).ToList();
var holdStartMap = new Dictionary<string, (object Note, BmsNote BmsNote)>();

// 2. perfectSample을 직접 비교하여 더 정확한 매칭
foreach (var holdStartBms in holdStartBmsNotes)
{
    var expectedPerfectSample = (int)(holdStartBms.Time * SAMPLE_RATE);
    
    foreach (var noteObj in noteList)
    {
        var perfectSample = perfectSampleField?.GetValue(noteObj);
        if (perfectSample != null && (int)perfectSample == expectedPerfectSample)
        {
            // Lane, IsLeft, noteTypeID도 확인하여 정확히 매칭
            // ...
            holdStartMap[key] = (noteObj, holdStartBms);
            break;
        }
    }
}

// 3. 끝 노트 매칭 (시간 허용 범위 0.05초)
foreach (var holdEnd in holdEndNotes)
{
    if (holdStartMap.TryGetValue(searchKey, out holdStart))
    {
        // Note가 null인 경우 새로 생성하고 noteList에 추가
        if (holdStart.Note == null)
        {
            holdStart.Note = createNoteCreateData(holdStart.BmsNote);
            noteList.Add(holdStart.Note); // ⚠️ 중요: noteList에 추가
        }
        
        // 시간 차이 확인 (0.05초 허용)
        if (Math.Abs(holdStart.BmsNote.Time + holdStart.BmsNote.Duration - holdEnd.Time) < 0.05f)
        {
            NoteProcessorHelper.AddEndNoteToConnectNodeArray(...);
        }
    }
    else
    {
        // 범위 검색: 허용 범위 내에서 가장 가까운 시작 노트 찾기
        // ...
    }
}
```

**핵심 포인트:**
- **`allBmsNotes`에서 직접 찾기**: `getBmsNoteFromNoteCreateData`의 부동소수점 오차 문제 회피
- **`perfectSample` 직접 비교**: 기존 노트를 정확히 매칭하여 중복 생성 방지
- **`noteList`에 노트 추가**: 매칭되지 않은 노트를 새로 생성할 때 `noteList`에 추가하여 게임에 반영
- Dictionary 기반 O(1) 검색으로 성능 최적화
- 키: `Lane + IsLeft + EndTime` (소수점 3자리까지)
- **BPM 기반 동적 시간 허용 범위**: 기준 BPM 120일 때 약 0.05초, 최소 0.02초 ~ 최대 0.15초 (`CalculateTimeTolerance(baseBpm)`)
- **범위 검색 추가**: 정확한 키 매칭 실패 시 허용 범위 내에서 가장 가까운 노트 찾기 (같은 레인·같은 방향(IsLeft)만)
- 끝 노트의 `directionIndex`를 `CENTER_MIDDLE`로 설정
- 끝 노트의 `laneLeftRightID`, `subLaneID`, `noteTypeID`를 시작 노트와 동일하게 설정

## 알려진 문제 및 해결 방법

### 문제 1: 홀드 끝 노트 매칭 실패

**증상:**
```
[HoldNoteProcessor] 홀드 끝 노트 매칭 실패: Lane=X, IsLeft=Y, Time=Z
```

**원인:**
- BMS 파일에서 홀드 시작(02)과 끝(19)이 제대로 매칭되지 않음
- 같은 레인에 여러 홀드가 겹쳐 있는 경우
- BPM 변화로 인한 시간 계산 오차

**해결 방법 (2024년 최신 버전):**
1. **자동 해결됨**: 개선된 매칭 로직으로 대부분의 경우 자동으로 해결됩니다
   - `allBmsNotes`에서 직접 찾기로 부동소수점 오차 문제 해결
   - `perfectSample` 직접 비교로 정확한 매칭
   - 시간 허용 범위 0.05초로 확대
   - 범위 검색으로 근사치 매칭 지원
2. BMS 파일 확인: 홀드 시작(02)과 끝(19)이 같은 레인에 있는지 확인
3. 디버그 모드 활성화: JSON 덤프로 실제 노트 데이터 확인
4. 로그 확인: `[HoldNoteProcessor]` 로그에서 매칭 과정 확인

### 문제 2: 홀드 끝 노트 시간 불일치

**증상:**
```
[HoldNoteProcessor] 홀드 끝 노트 시간 불일치: Expected=X, Actual=Y
```

**원인:**
- BPM 변화로 인한 시간 계산 오차
- Tick → Time 변환 과정에서의 부동소수점 오차

**해결 방법:**
1. BPM 변화 확인: BMS 파일의 BPM 변화가 정확한지 확인
2. 시간 오차 허용 범위 조정: `0.01f` 값을 필요에 따라 조정
3. BPM 계산 로직 개선: `BmsParser.CalculateTime()` 메서드 확인

### 문제 3: 홀드 끝 노트가 플릭으로 잘못 생성됨 ⚠️

**증상:**
- 홀드 끝 노트가 `connectNodeDataArray`에 넣지 않고 메인 노트 배열에 그대로 생성됨
- 메인 노트 배열에 생성된 끝 노트가 플릭 노트로 표시됨
- JSON 덤프에서 메인 노트 배열에 `noteTypeID`가 `"Flick"`인 노트가 있음
- 홀드 노트가 제대로 작동하지 않음 (끝 노트가 별도 플릭 노트로 생성됨)

**원인:**
1. **BMS 파일 오류**: 홀드 끝 노트가 `19`가 아닌 플릭 값(`03-0A`)으로 입력됨
   - `BmsParser.GetNoteType()`에서 `03-0A`는 `NoteType.Flick`으로 인식됨
   - `BmsNoteConverter.ConvertBmsNotesToNoteCreateData()`에서 `bmsNote.Type == NoteType.HoldEnd` 체크를 통과하지 못함
   - 따라서 `holdEndNotes`에 추가되지 않고 일반 노트처럼 메인 배열에 추가됨
   - `NoteCreateDataBuilder.CreateNoteCreateData()`에서 `bmsNote.Type`이 `Flick`이므로 플릭으로 생성됨

2. **파싱 및 변환 과정**
   ```csharp
   // 1단계: BmsNoteDataParser.GetNoteType()
   if (value == 0x19) return NoteType.HoldEnd;    // 정상: 19 → HoldEnd
   
   // 하지만 19가 아닌 03-0A가 사용되면:
   if (value >= 0x03 && value <= 0x0A)  // 플릭 범위
   {
       direction = GetFlickDirection(value);
       return NoteType.Flick;  // ⚠️ 플릭으로 잘못 인식됨!
   }
   
   // 2단계: BmsNoteConverter.ConvertBmsNotesToNoteCreateData()
   if (bmsNote.Type == NoteType.HoldEnd)  // ⚠️ Flick이므로 이 체크를 통과하지 못함!
   {
       holdEndNotes.Add(bmsNote);
       continue;  // 메인 배열에 추가하지 않음
   }
   
   // 3단계: 일반 노트처럼 처리됨
   var noteCreateData = Builders.NoteCreateDataBuilder.CreateNoteCreateData(bmsNote);
   // bmsNote.Type이 Flick이므로 플릭으로 생성됨!
   noteList.Add(noteCreateData);  // ⚠️ 메인 배열에 플릭 노트로 추가됨
   ```

3. **결과**
   - 홀드 끝 노트가 `connectNodeDataArray`에 들어가지 않음
   - 대신 메인 노트 배열에 플릭 노트로 생성됨
   - 게임에서 홀드 노트가 제대로 작동하지 않음

4. **BMS 에디터에서 잘못된 WAV 인덱스 사용**
   - 홀드 끝 노트에 플릭용 WAV 인덱스(`03-0A`)를 사용한 경우

**해결 방법:**

1. **BMS 파일 확인 및 수정**
   - BMS 에디터에서 홀드 끝 노트가 `19` 값으로 설정되어 있는지 확인
   - 플릭 값(`03-0A`)이 홀드 끝 노트에 사용되지 않았는지 확인
   - 올바른 형식:
     ```
     #00111: 0002000000000000  // measure 1, 채널 11, 홀드 시작(02) ✅
     #00211: 0019000000000000  // measure 2, 채널 11, 홀드 끝(19) ✅
     ```
   - 잘못된 형식 (플릭으로 인식됨):
     ```
     #00111: 0002000000000000  // measure 1, 채널 11, 홀드 시작(02) ✅
     #00211: 0003000000000000  // measure 2, 채널 11, 값 03 → 플릭으로 인식! ❌
     #00211: 0007000000000000  // measure 2, 채널 11, 값 07 → 플릭으로 인식! ❌
     ```
   - BMS 에디터에서 WAV 인덱스 확인:
     - 홀드 시작: WAV 인덱스 `01` (BMS 값 `02`) 사용
     - 홀드 끝: WAV 인덱스 `19` (BMS 값 `19`) 사용 ⚠️ **중요!**
     - 플릭 WAV 인덱스(`02-09`, BMS 값 `03-0A`)는 홀드 끝에 사용하지 않음

2. **JSON 덤프로 확인**
   - `note/bms_BeforeCreateAllNote_*.json` 파일 확인
   - **정상적인 경우:**
     ```json
     [
       {
         "noteTypeID": "Hold",
         "perfectSample": 100000,
         "connectNodeDataArray": [
           {
             "noteTypeID": "Hold",  // ✅ 끝 노트가 connectNodeDataArray에 있음
             "directionIndex": "CENTER_MIDDLE",
             "perfectSample": 200000
           }
         ]
       }
     ]
     ```
   - **잘못된 경우 (끝 노트가 메인 배열에 플릭으로 생성됨):**
     ```json
     [
       {
         "noteTypeID": "Hold",
         "perfectSample": 100000,
         "connectNodeDataArray": []  // ❌ 빈 배열! 끝 노트가 없음
       },
       {
         "noteTypeID": "Flick",  // ❌ 끝 노트가 메인 배열에 플릭으로 생성됨!
         "directionIndex": "LEFT_MIDDLE",  // 플릭 방향이 설정됨
         "perfectSample": 200000  // 홀드 끝 시간과 동일
       }
     ]
     ```
   - **확인 포인트:**
     - 홀드 시작 노트의 `connectNodeDataArray`가 비어있지 않은지 확인
     - 메인 노트 배열에 홀드 끝 시간과 동일한 시간에 플릭 노트가 있는지 확인
     - 플릭 노트의 `perfectSample`이 홀드 끝 시간과 일치하는지 확인

3. **로그 확인**
   - `[BmsParser]` 로그에서 홀드 끝 노트 타입 확인:
     ```
     [BmsParser] 노트 파싱: Channel=11, Value=19, Type=HoldEnd  // ✅ 정상
     [BmsParser] 노트 파싱: Channel=11, Value=03, Type=Flick  // ❌ 잘못됨!
     ```
   - `[HoldNoteProcessor]` 로그에서 끝 노트 처리 확인
   - `[NoteCreateDataBuilder]` 로그에서 생성된 끝 노트 타입 확인:
     ```
     [NoteCreateDataBuilder] 노트 생성: noteTypeID=Hold  // ✅ 정상
     [NoteCreateDataBuilder] 노트 생성: noteTypeID=Flick  // ❌ 잘못됨!
     ```

4. **코드 수정 (임시 해결책)**
   - 만약 BMS 파일을 수정할 수 없다면, `BmsNoteConverter.ConvertBmsNotesToNoteCreateData()`에서
   - 특정 조건(예: 홀드 시작 노트와 시간이 가까운 플릭 노트)을 체크하여
   - `holdEndNotes`에 추가하도록 수정할 수 있음
   - 하지만 이는 권장하지 않음 (BMS 파일 수정 권장)

**예방 방법:**
- BMS 에디터에서 홀드 끝 노트 생성 시 올바른 WAV 인덱스(`19`) 사용
- BMS 파일 검증 도구 사용하여 잘못된 노트 타입 감지
- 홀드 시작(02)과 끝(19)이 올바른 쌍으로 사용되는지 확인

### 문제 4: 홀드 노트가 게임에서 제대로 표시되지 않음

**증상:**
- 홀드 노트가 화면에 나타나지 않음
- 홀드 노트가 짧게 표시됨

**원인:**
- `connectNodeDataArray`에 끝 노트가 제대로 추가되지 않음
- 끝 노트의 `directionIndex`가 잘못 설정됨

**해결 방법:**
1. 디버그 모드 활성화: JSON 덤프로 `connectNodeDataArray` 확인
2. 끝 노트 확인: `directionIndex`가 `CENTER_MIDDLE`로 설정되었는지 확인
3. 로그 확인: `[HoldNoteProcessor]` 로그 메시지 확인

### 3. 로그 확인
- `[BmsParser]` 로그에서 홀드 끝 노트 타입 확인:
  ```
  [BmsParser] 노트 파싱: Channel=11, Value=02, Type=Hold  // ✅ 정상
  [BmsParser] 노트 파싱: Channel=11, Value=19, Type=HoldEnd  // ✅ 정상
  [BmsParser] 노트 파싱: Channel=11, Value=03, Type=Flick  // ❌ 잘못됨! (끝 노트가 플릭으로 인식)
  ```
- `[BmsNoteConverter]` 로그에서 끝 노트 처리 확인:
  ```
  [BmsNoteConverter] BMS 노트 변환 시작: X개
  // ✅ 정상: HoldEnd 타입은 holdEndNotes에 추가되고 메인 배열에 추가되지 않음
  // ❌ 잘못됨: Flick 타입은 holdEndNotes에 추가되지 않고 메인 배열에 추가됨
  ```
- `[HoldNoteProcessor]` 로그에서 끝 노트 처리 확인:
  ```
  [HoldNoteProcessor] 홀드 끝 노트 처리 시작: 0개  // ❌ 0개면 끝 노트가 holdEndNotes에 없음!
  [HoldNoteProcessor] 홀드 끝 노트 처리 시작: 1개  // ✅ 정상
  [HoldNoteProcessor] noteList 개수: 358개  // noteList 크기 확인
  [HoldNoteProcessor] 홀드 시작 노트 맵 생성 완료: 3개 (noteList 매칭: 2개, 매칭 실패: 1개)  // 매칭 상태 확인
  [HoldNoteProcessor] ✓ 홀드 끝 노트 매칭 성공: Time=X, StartTime=Y, Duration=Z  // ✅ 정상
  [HoldNoteProcessor] ✓ connectNodeDataArray에 끝 노트 추가 성공: 최종 길이=1  // ✅ 정상
  [HoldNoteProcessor] 끝 노트 상세: noteTypeID=Hold, perfectSample=X, directionIndex=CENTER_MIDDLE  // 끝 노트 정보 확인
  [HoldNoteProcessor] 홀드 끝 노트 매칭 완료: 성공=3개, 실패=0개  // 최종 결과 확인
  ```
- `[NoteCreateDataBuilder]` 로그에서 생성된 노트 타입 확인:
  ```
  [NoteCreateDataBuilder] 노트 생성: noteTypeID=Hold  // ✅ 정상 (시작 노트)
  [NoteCreateDataBuilder] 노트 생성: noteTypeID=Hold  // ✅ 정상 (끝 노트, connectNodeDataArray에)
  [NoteCreateDataBuilder] 노트 생성: noteTypeID=Flick  // ❌ 잘못됨! (끝 노트가 메인 배열에 플릭으로 생성)
  ```
- **핵심 확인 사항:**
  - `[HoldNoteProcessor] 홀드 끝 노트 처리 시작: 0개`가 나오면 끝 노트가 `holdEndNotes`에 없음
  - 메인 노트 배열에 플릭 노트가 생성되었는지 확인
  - 플릭 노트의 시간이 홀드 끝 시간과 일치하는지 확인
- `[HoldNoteProcessor]` 로그에서 끝 노트 처리 확인:
  ```
  [HoldNoteProcessor] 홀드 끝 노트 처리 시작: X개
  [HoldNoteProcessor] connectNodeDataArray에 끝 노트 추가 완료  // ✅ 정상
  ```

## 주요 개선 사항 (2024년 최신 버전)

### 1. 채널 필터링 개선
- **이전**: 채널 11-18만 필터링 (채널 16 누락 가능)
- **개선**: `ChannelToLaneMap`을 사용하여 모든 노트 채널(11, 12, 14, 15, 16, 18) 포함

### 2. 매칭 로직 개선
- **이전**: `getBmsNoteFromNoteCreateData`를 사용하여 `noteList`에서만 찾기 (부동소수점 오차로 매칭 실패 가능)
- **개선**: `allBmsNotes`에서 직접 홀드 시작 노트를 찾아서 `noteList`와 매칭
- **개선**: `perfectSample`을 직접 비교하여 정확한 매칭 및 중복 생성 방지

### 3. 노트 생성 및 추가
- **이전**: 매칭되지 않은 노트를 새로 생성하지만 `noteList`에 추가하지 않음 (게임에 반영 안 됨)
- **개선**: 새로 생성한 노트를 `noteList`에 추가하여 게임에 반영

### 4. 시간 허용 범위 확대
- **이전**: 0.01초 이내만 매칭
- **개선**: 0.05초로 확대하여 부동소수점 오차 대응

### 5. 범위 검색 추가
- **이전**: 정확한 키 매칭 실패 시 매칭 실패로 처리
- **개선**: 허용 범위 내에서 가장 가까운 시작 노트를 찾아서 매칭

### 6. 끝 노트 필드 설정 보강
- **개선**: 끝 노트의 `laneLeftRightID`, `subLaneID`, `noteTypeID`를 시작 노트와 동일하게 설정
- **개선**: Boolean 필드들(`isLast`, `isCritical` 등) 설정

### 7. 페어리 노트 (매칭 규칙은 홀드와 다름)
- **MatchFairyNotes()**: 홀드와 달리 **같은 `IsLeft`(왼쪽/오른쪽) 그룹 내에서만** 시작(11-18)과 끝(1A-1B) 매칭. **레인 제한 없음**(레인을 넘나드는 슬라이드 연결 지원). Tick 오름차순, FIFO 큐로 1:1 연결. 상호 참조 `StartNote`/`EndNote` 설정.
- **ProcessFairyEndNotes()**: `StartNote` 참조로 시작 노트 찾기(우선). 없으면 폴백으로 (Time, Lane, IsLeft) + 허용오차 0.01초. 끝 노트의 `directionIndex`는 1A/1B에 따라 createNoteCreateData에서 설정·유지(Helper에서 덮어쓰지 않음). `turnDireciton`은 시작 노트와 동일 복사.
